# Portfolio Audit: ch.sh

> Benchmark analysis, not a fix list. Studying what places this portfolio in the top tier and where it falls short of even higher-tier work.

## Overall Impression

This is a top-tier portfolio that occupies an entirely different lane from benji.org. Where benji.org is editorial restraint, ch.sh is **systems infrastructure**. The homepage is a personal index — bio, a 22-screen image gallery launched from a single "View work" button, three feature shortcuts, six project rows, four art rows, five connect rows, a 30-day commit graph with 94 commits visible, a live clock, weather, and a Korean signature character. There is more dense identity-signal per pixel here than on any portfolio in the reference set. The cost is cognitive density — a hiring manager has to read it like a dashboard, not skim it like a magazine. The reward is that he has demonstrated builder-velocity AND craft AND breadth AND AI fluency on a single fold. A senior designer who builds at this rate is the rarest profile in 2026.

## What's Working (the moves, examined)

**The commit graph as portfolio surface.** A 30-day bar chart of GitHub commits, with daily counts and which repos were touched ("Apr 14 / 14 commits / portfolio, skills, ramp"). The total — "94 commits / Last 30 days" — sits as a metric in the upper-right block. This is a builder signal you cannot fake. Anyone can write "I ship a lot." Almost no one shows the receipts at this granularity. The graph is computed/rendered live, not a static image — the data is real. This single element does the work that two case studies would normally do.

**The /skills and /stack as separate IA surfaces, not buried.** The top nav is THREE items: home (logo), /skills, /stack. Not "About / Work / Contact." Charles has decided that what he uses (stack) and what skills he runs (skills) ARE the portfolio surfaces. /stack is 155+ curated resources across 16 categories. /skills is 23 Claude Code skills (2 self-authored, 21 installed). This is portfolio-as-system: he is positioning himself as someone who curates AND builds the tooling layer, not just the artifact layer.

**Live clock + weather + git commit hash + Korean signature in the footer.** Four humanity touches that compound. Most portfolios have zero. The git commit hash (linked to the actual GitHub commit) is the strongest of the four — it broadcasts "this site is a public git repo, my deployment is visible, my work is auditable." This is a builder-trust signal that no amount of case-study writing could equal.

**The "Index 22" modal gallery.** The Freighter case study lives behind a single "View work" button on the homepage. Clicking opens an in-place gallery with "Index 22" — a numbered count of 22 wallet screens, each labeled (Home / Past transactions / Wallets / Discover / Discover detail / In-app browser / Transaction details / Malicious transaction warning / Review transaction / etc.). Modal style, single-image-at-a-time, breadcrumb at the bottom ("Freighter / Home"). This is a unique presentation format — not a case study, not a thumbnail grid, but a *gallery walkthrough* in the format of a designer's portfolio review. The numbered "Index 22" is the cue that this is a complete UI flow review, not a marketing piece.

**Dense iconography across all project rows.** Every project, art piece, and connect link has a hand-crafted SVG icon. The icons are not stock — each one is bespoke (ElevenRooms is a four-bar audio-spectrum mark, RGB is a stylized cube, Rat Labs is a pixelated face, Noundation/Noundation UI share a fork-glyph variant). The icon library alone represents 15+ small designs. This is an *icon-system designer's* portfolio — the discipline of making and managing icons is itself the signal.

**One-line descriptors that work as poetry.** "Flagship Stellar wallet." "List curation tool." "Stellar developer sandbox." "ASCII Soundscapes." "Onchain identity framework." "Onchain product studio." "Community design system." "Daily brand remixes." "Zorb profile collection." Each is 2–5 words, declarative, no marketing fluff. Compare to portfolio descriptors like "A revolutionary platform that empowers users..." — opposite tone.

**Color discipline AS systems thinking.** Project list icons are monochrome (black on white). Art piece icons are full-color (purple/pink/yellow/blue). The color separation by ROLE — not by visual variety — is design-system thinking applied to the portfolio. "Color goes where the work is colorful (Art). Color does not go where the work is structural (Projects)." This is the same logic that powers a real design system token architecture.

**The bio names AI tools by their proper-noun identity.** "Writing code through conversation and wiring agents together (Claude Code, OpenClaw, Paperclip)." Three named tools. Two of them (OpenClaw, Paperclip) are obscure-enough that a reader can tell they're insider names — the kind of references that signal "I'm operating at the bleeding edge." Compare to "uses AI tools" — that's how junior designers describe what Charles does. The proper-noun naming is a positioning move.

**The 신 character in the footer.** Charles's Korean surname character, used as a signature mark next to "© 2026". This is heritage signal at near-zero implementation cost. It tells you something about him in one character that a "About me" paragraph would take 50 to convey.

**"Software Designer" as a chosen title.** Not "Product Designer," not "UX Designer," not "Design Engineer." "Software Designer" is a deliberate positioning move — it implies fluency at both the code and design layers without requiring the qualifier "design engineer." Combined with the line "Writing code through conversation and wiring agents together," the title is internally consistent — he writes code, but does so through agents, so he is neither "an engineer who designs" nor "a designer who codes." He is positioning a new category.

**The Featured / Projects / Art tri-section is editorial taxonomy.** Featured = paid/employer work currently representing him (Stellar ecosystem — Freighter, Snack, Laboratory). Projects = side projects with live URLs. Art = NFT/onchain art pieces. The three categories represent three different proof types: employer trust, builder velocity, creative practice. A reader can decide which axis to evaluate him on.

## Where it falls short of even higher-tier portfolios

**The right column dies below the bio block.** This is the genuine craft issue. On desktop the first viewport has bio + view-work-card + commit-graph as a two-column composition. Once the user scrolls past the bio (~600px), the right column becomes white space and the left column carries every remaining row alone. That's roughly 1200px of vertical real-estate on the right that does NOTHING. A higher-tier execution would either (a) extend the two-column rhythm with a "Latest writing" / "Now playing" / "Currently building" surface on the right, or (b) center everything below the bio so the asymmetry is intentional. Right now it reads as "the two-column layout ran out of content."

**The Featured items don't show the work.** Freighter, Snack, Laboratory — three Stellar-ecosystem projects that are the strongest signal of Charles's recent work. They're presented as icon + name + one-line description, with NO preview image, NO link arrow, NO visible interaction state. They're buttons (per the DOM — `expanded=false` aria attribute) that probably open the gallery modal — but the affordance is invisible to a first-time visitor. A user has to click "View work" on the upper-right card to even discover that the gallery exists. The Featured row should be the gallery's entry point, not a separate card 600px away.

**The Laboratory entry is greyed out.** Both lines of text are visibly muted compared to Freighter and Snack. This is presumably "coming soon" or "not yet linked" — but without a label, it just looks like a UI bug. A subtle "soon" tag or italic-treatment would convert the bug-look into a deliberate choice.

**Project lineage is multi-counted.** RGB appears as a Project (rgb.fun). RGB Signatures appears as Art. RGB Friends appears as Art. That's three RGB entries on the same surface. Similarly, Noundation and Noundation UI are sibling products of the same brand. These multi-entries inflate the project count without adding distinct evaluation signal. A reader trying to apply the removal test ("if I removed this, would the average rise?") would mark at least 2 of those entries as redundant. The reverse case for keeping them — that each is genuinely a different artifact — only holds if the gallery clarifies the difference. At the list-row level, they look like padding.

**The "Index 22" gallery has no narrative.** The 22 screens are labeled (Home / Past transactions / etc.) but there is no prose explaining the design decisions, the constraints, the problems solved, or the outcomes. This is the same trade-off benji.org makes (work over narrative) — but benji.org backs each work with a long-form writing entry. Charles's gallery is the entire work surface, and it carries the entire narrative load via labels alone. A user evaluating "did Charles design this well?" can see the screens; a user evaluating "what was the *design thinking* behind these screens?" gets nothing. The gallery makes him look like an excellent visual designer but undersells him as a systems thinker.

**No long-form writing surface.** Charles has 23 Claude Code skills (per /skills), 155+ curated stack resources, a 94-commit-per-month build rate, and multiple shipped products. There is zero long-form essay or article on the site. This is one of the very few areas where benji.org is more developed (Liveline + Morphing icons + Annotating for agents + Family Values are essays). Charles has the corpus to write — the skills are public IP, the stack is curated knowledge, the commits are visible — but he has chosen NOT to convert any of it into prose. The omission is striking. Either the format isn't yet built, or it's a deliberate "shipping > writing" choice.

**The /skills page has thin per-item presentation.** Each skill is name + one-liner + GitHub link. There are no examples of what the skill produces, no screenshots, no demos. The 21 installed skills read as a *what I have configured* list, not a *what I get out of these* showcase. For a portfolio whose central thesis is "I run agent tooling," the skills page is the weakest surface — it has the most opportunity to prove the thesis and the least execution. Compare against a writing-with-Claude-code post that EMBEDS the working component (benji.org's Morphing icons) — that single embedded demo is more compelling than the entire /skills page.

**The /stack page is also list-only.** 16 categories, 155+ items, each presented as favicon + name. No "why I use it" descriptions, no ratings, no "currently using vs trying," no personal tags. The page is a directory, which is useful — but it positions Charles as a curator of other people's tools rather than a builder of his own. A higher-tier execution would interleave "things I use" with "things I built" — every Fun Site entry would be linked to whichever of his own products inspired or competes with it.

**No mobile preview of the work.** The "View work" gallery shows mobile wallet screens (Freighter is a mobile app). But the gallery shows them at fixed size on a desktop — they don't ADAPT to mobile viewport. A user opening ch.sh on an iPhone gets the same mobile-screen renders zoomed to fit the actual mobile viewport. This is a small responsive miss — the work surface should be tested on real mobile, and Freighter's mobile-app design is the obvious place to show iOS-tier responsive craft.

**The /work page is the gallery; there's no project-specific URL.** Clicking "View work" opens an overlay; the URL probably changes to `/work` (verified — /work is the Freighter gallery). There's no /work/freighter or /work/snack or /work/laboratory — Freighter, Snack, and Laboratory are NOT actually deep-linkable. The "Index 22" experience is essentially monolithic — it's one gallery of Freighter screens. If Snack and Laboratory have their own galleries, they're not visible. This is a curation/IA limit: only one project gets the visual treatment, and the other two are signposted but not delivered.

## Non-obvious observations

1. **The site is positioned for hiring managers AND tool-builder peers.** The /skills page is the wrong page for a hiring manager — it's a tool list. But it's the RIGHT page for a fellow builder, who wants to know what tools Charles uses to be productive at 94 commits/month. Charles is splitting his portfolio across two audiences and serving each one a different surface. This is unusual and sophisticated.

2. **The commit graph normalizes velocity as a portfolio signal.** Most portfolios hide velocity (slow ones don't want to admit, fast ones don't want to brag). Charles shows it as a fact. The graph also reveals the *patterns* of work — bursts of 14 commits in a day, weekend zeros, sustained sub-runs. A hiring manager reading this graph can predict Charles's working rhythm before any interview.

3. **The "신 © 2026" + Chicago location + live clock = location-identity stack.** Three signals in the footer that compound: ethnicity (Korean character), city (Chicago, IL), time (live clock). A reader gets enough information to know who Charles is, where, and when — without an /about page. The whole identity stack costs ~25 characters of UI.

4. **The right-column View Work card has a teasing magenta dot peeking from the bottom of the image stack.** This is a deliberate composition — the card shows the top of one screen, the "View work" CTA in the middle, and a hint of color from the next screen below the gradient mask. It's saying "there's more underneath; click to see." Most preview cards just show one image; this one hints at depth. The dot is the only place magenta appears on the homepage — a tiny accent moment.

5. **The Featured row places employer projects above side projects.** Most portfolios put their flashiest side projects up top. Charles inverts this: paid employer work (Stellar ecosystem) is "Featured," then side projects in "Projects," then "Art." The ordering implicitly says "I take my employer work seriously enough to surface it first." This is unusual — most portfolios undervalue employer work because of NDA constraints, but Charles has clearly negotiated visibility.

6. **The site is built as a Next.js app with /_next/image optimization (seen in the image URLs).** The technical stack is visible from the markup — this is a working Next.js app, not a static-site export. Combined with the linked git commit hash, the site itself is also a builder-portfolio piece: it's a working production app that he ships to himself.

7. **The "View work" gallery's images are mobile screens, not desktop UI.** Freighter is presented entirely as iPhone-sized renders. This is an *iOS-craft signal* — Charles is showing fluency at the mobile-design layer specifically. Most blockchain wallet portfolios show wallet UI as desktop+mobile composites; he chose mobile-only. The implicit message: "iOS craft is what I want to be evaluated on for this project."

8. **The "1:52:35 PM" clock has no timezone indicator.** Charles is in Chicago. The clock is presumably local. A reader from a different timezone sees the time but has to infer the location from the footer. This is the OPPOSITE of benji.org's clock, which spells out "11:45am in Los Angeles, California." Charles's clock is *more* minimal and *less* explanatory — it bets on the reader to figure it out.

## The Hiring Manager Gut Check

Would I ping him? Yes, instantly. But I'd hire him for a different role than I'd hire benji.org's author. Charles reads as a *systems person* — someone who would set up the team's tooling, build the agent layer, ship 5x what the average designer ships, and bring an unusual combination of crypto + AI + design-engineering fluency. Benji reads as a *taste arbiter* — someone who would set the bar for craft and write the document that explains why. Both are senior; they're different shapes of senior. For a fast-shipping startup that needs an in-house Claude-Code expert who can ALSO design well, Charles is the higher-signal pick.

## Distilled lessons for a benchmarking designer

1. **A live commit graph is a builder signal that 99% of portfolios miss.** If you ship, show it. The graph is more credible than any case study.

2. **/skills and /stack as separate pages is a category bet.** Charles has decided that *what you use* and *what you can run* are first-class portfolio surfaces. This is a 2026 bet: if AI/agent fluency becomes the dominant hiring signal, his architecture is ahead of the curve.

3. **Bespoke icon library is the dense-portfolio's typography.** Where benji.org leans on typography for identity, Charles leans on a custom icon system. 15+ tiny SVGs across the site, every one different, all unified in style. This is a system signal — he's not just shipping one product, he's shipping a coherent visual language.

4. **Footer-as-status-bar.** Live clock + weather + city + git hash + signature character. Five humanity signals in a footer that takes ~40px of vertical space. Most portfolios have a copyright line; Charles has a status bar.

5. **Modal gallery > separate case study pages, when the work is visual.** Charles's "Index 22" gallery is faster to consume than a full case study would be. The trade-off is no narrative — but for mobile-UI design work, the screens speak for themselves. Choose the format that fits the work, not the template.

6. **Color discipline by role, not by variety.** Project icons are monochrome (because the work is structural). Art icons are full-color (because the work is colorful). The rule is *internal to the system* — color goes where the work warrants it.

7. **Naming specific tools elevates the AI-fluency signal.** "Claude Code, OpenClaw, Paperclip" is the move. Not "AI tools," not "Claude," not "agents in general." The specificity is the credibility.

8. **A dense homepage is a hiring filter.** Charles's homepage is dense enough that a casual reader bounces. The audience who LIKES dense homepages — engineers, technical hiring managers, founders — is the audience he's filtering for. Restraint is one strategy; deliberate density is another.
