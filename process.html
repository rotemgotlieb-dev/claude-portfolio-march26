How I work with AI.

I shipped three products this year with an AI building beside me. Not autocomplete, a real operating model. The split is clear. It brings velocity and breadth. I bring taste and the bar. Here is how the work actually happens.

The fork model

Every decision is a fork.

The AI does not wait to be told, and it does not run loose. For anything that matters it lays out a fork: the real options, a recommendation, and the tradeoff between them. I lock one.

Sometimes I take the recommendation. Sometimes I overrule it. On the Pulse case study spine I chose the riskier option B over the safe A, because a signature signal was worth the saturation risk. The AI then builds what I locked, not what it preferred.

One rule sits above all of it. No production code until the mockup is approved. Pulse's home screen went through four mockups before a single real line shipped, and each round of redlines caught something the words had hidden. A hover arrow colliding with wrapped text. A tour spotlight ten pixels off.

Tests versus truth

Passing tests are not a working product.

This is the lesson all three products taught me, each one on its own. Pulse had 116 tests green and the type checker clean while a compare panel sat there contradicting the heatmap right beside it. Ghost shipped headings rendered white on white, invisible, every test still passing.

The tests asserted that things existed. They could not see that things were wrong. Mechanical gates check syntax and contracts. They are blind to whether the pixels are right.

How we verify

So we verify like skeptics.

If green is not enough, something has to look at the rendered thing. Two habits close the gap.

First, the AI reviews its own work adversarially. Independent reviewers raise findings, then separate refuters try to disprove each one before it counts. On Pulse, two passes confirmed sixteen real bugs with zero false positives, while every test stayed green. Structured skepticism sees what a checkmark cannot.

Second, nothing I have not watched gets shown to you. Every animation on this site was checked frame by frame in a real browser before it shipped. A deploy is not live until an anonymous request returns it correctly, not when the dashboard says the build passed.

Memory as a system

Memory is infrastructure, not notes.

The method only compounds if the AI remembers. Early on that meant one file it read every session. Thirteen hundred lines, around forty thousand tokens, most of it irrelevant to the task at hand.

I re-architected it into a small always loaded core, topical rules pulled in only when the matching work starts, and a checklist of past failures read only at review time. Session start cost dropped by a factor of sixty to eighty. Every bug a review catches becomes one line in that checklist, so the next session never re-debugs it.

This is the same instinct as a design token. Solve a thing once, encode the solution as infrastructure, and stop solving it.

Show the seams

The seams are the proof.

None of these launches were clean, and I left the evidence in. Ghost's first deploy returned an error on every route behind a login wall, then served a dead site because a framework preset guessed wrong, then stalled because an old email address looked to the platform like a paywall. Pulse's entire compare feature once vanished below a certain screen width, because its only output lived in a column that could hide.

Most case studies claim a smooth line from idea to launch. This one shows the debugging. A method is only worth trusting when you can see where it broke and how it recovered.
