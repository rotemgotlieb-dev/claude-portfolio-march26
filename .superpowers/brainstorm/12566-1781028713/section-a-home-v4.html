The fix: the spotlight and tooltip are now DOM children of .priority itself, which is the target of step 1. The spotlight uses inset:-4px to trace the priority strip's outer bounds + a 4px breathing ring. The tooltip uses top: calc(100% + 14px) to sit 14px below the strip. Whatever happens to the priority strip's actual rendered position (viewport resize, content wrap, padding change, anything) — the overlay follows it. Zero hardcoded coordinates.

Production implementation note (for when we build the Tour component in React): each target element on the page gets a data-tour-target attribute (priority / tabs / layers / friction). The Tour component reads document.querySelector('[data-tour-target="X"]').getBoundingClientRect() per step, positions the spotlight + tooltip in a portal, and uses ResizeObserver to re-measure on layout shift. Step transitions animate the spotlight rect from old to new (300ms cubic-bezier(0.32, 0.72, 0, 1)). For tooltip positioning with auto-placement + viewport collision, we'll use @floating-ui/react. Standing rule added to pulse/learnings.md.
